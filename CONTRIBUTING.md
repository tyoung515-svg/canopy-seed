# Contributing to Canopy Seed

Canopy Seed was built by AI agents coordinated by humans. You can contribute the same way — with or without AI assistance.

---

## For Human Contributors

### Setting Up a Development Environment

1. **Clone the repository** and navigate to the Canopy Seed folder.

2. **Create a Python virtual environment:**
   ```bash
   python -m venv venv
   ```

3. **Activate the venv:**
   - **Windows:** `venv\Scripts\activate`
   - **Linux/Mac:** `source venv/bin/activate`

4. **Install dependencies:**
   ```bash
   pip install -r requirements.txt
   pip install -r requirements-dev.txt  # Includes pytest, black, etc.
   ```

5. **Start the server:**
   ```bash
   python agent.py
   ```

6. **Open `http://localhost:7821` in your browser** to see the UI.

### Code Style

Canopy doesn't enforce a strict formatter (no Black auto-formatting pipeline). Instead:

- **Readable names.** Use clear variable names (`conversation_id`, not `cid`).
- **Consistent indentation.** 4 spaces (Python standard).
- **Docstrings for functions.** See "Code Comments" below.
- **Type hints** for function signatures (helps with IDE autocomplete).
- **One logical change per commit.** Don't mix refactoring with feature work.

### Running the Test Suite

```bash
pytest tests/ -v
```

This runs all tests in the `tests/` directory. The `-v` flag shows detailed output.

Run a specific test file:
```bash
pytest tests/test_context_builder.py -v
```

Run a specific test:
```bash
pytest tests/test_context_builder.py::test_simple_conversation -v
```

All tests must pass before you submit a PR. If you add a new feature, add a test for it.

### Submitting a Pull Request

1. **Create a branch** for your feature or fix:
   ```bash
   git checkout -b feature/my-feature
   ```

2. **Write code and tests.** Test-first is encouraged.

3. **Run the full test suite** to make sure you didn't break anything:
   ```bash
   pytest tests/ -v
   ```

4. **Commit with a clear message:**
   ```bash
   git commit -m "Add support for image uploads in Big Brain conversation"
   ```

5. **Push to your fork:**
   ```bash
   git push origin feature/my-feature
   ```

6. **Open a pull request** on GitHub with:
   - Clear title ("Add image support to Big Brain")
   - Description of what changed and why
   - Link any related issues ("Closes #42")

**PR review will check:**
- Tests pass
- Code follows style guidelines
- No security issues (especially for shell exec, file write, web fetch)
- Docstrings are complete
- No orphan TODOs without issue references

## Contributing with AI Agents

Canopy Seed was built using its own methodology. You can extend it the same way.

### The Pattern

1. **Read the architecture.** Before asking an AI to build something, understand:
   - `ARCHITECTURE.md` — system design
   - `DECISIONS.md` — trade-offs made
   - The code files you're modifying

2. **Open an issue describing your feature.** Include:
   - What problem you're solving
   - How it should work (from a user perspective)
   - Any constraints (performance, compatibility)
   - Example usage

3. **Write a spec file** (like the ones in `docs/agent-specs/`) that gives an AI agent:
   - **Clear ownership:** Specific files the agent will modify. List files it will NOT touch.
   - **Exact function signatures:** If the agent needs to add a function, define the signature (parameters, return type).
   - **Test requirements:** What tests must pass?
   - **Example inputs/outputs:** Show what the feature should do.

4. **Run the agent** against your spec file (using Claude Code or similar).

5. **Review the output yourself** before merging:
   - Does the code follow the style guidelines?
   - Are there docstrings for all functions?
   - Are security-critical code paths commented?
   - Do all tests pass?
   - Did the agent touch files it wasn't supposed to?

6. **Submit a PR** with your agent's code. In the PR description, note that it was generated by an AI agent:
   ```
   Generated by Claude Code with spec: docs/agent-specs/AGENT_FEATURE_SPEC.md
   ```

We don't hide AI contributions. We ask you not to either.

### Example Spec File

Here's what a good spec file looks like:

```markdown
# Feature Spec: Add Slack Integration

## Mission
Add the ability to export Canopy projects directly to Slack channels.

## Ownership
**Files to create/modify:**
- `skills/slack.py` (NEW — implements SlackSkill)
- `tools/slack_client.py` (NEW — handles Slack API calls)
- `core/router.py` (MODIFY — register !slack command)
- `tests/test_slack_skill.py` (NEW — tests)

**DO NOT TOUCH:**
- `canopy-ui/` (frontend)
- `core/context_builder.py` (core context logic)
- Any JOAT-specific files

## Requirements

### New Skill: SlackSkill
File: `skills/slack.py`

```python
class SlackSkill(Skill):
    name = "slack"
    triggers = ["slack"]
    
    async def execute(self, args: str, update, context) -> str:
        """
        Commands:
        !slack send <channel> <message>    — send message to Slack channel
        !slack export <project_id>         — export project context to Slack thread
        """
```

### Slack Config
Add to `config/config.yaml`:
```yaml
slack:
  enabled: false
  bot_token: ""
  signing_secret: ""
```

### Tests
- Test successful message send
- Test invalid channel names
- Test project export formatting
- Test missing credentials (error handling)

## Success Criteria
- `pytest tests/test_slack_skill.py -v` — all pass
- !slack send #general "test" produces no errors
- Project export formats correctly for Slack
```

## Adding a Skill

Skills are the extensible command system in Canopy Seed. Here's how to add one.

### 1. Use `skills/example_skill.py` as a Template

```python
from skills.base import Skill
import logging

logger = logging.getLogger(__name__)

class ExampleSkill(Skill):
    name = "example"
    description = "This skill does example things."
    triggers = ["example", "ex"]  # Command names that trigger this skill
    enabled = True
    
    async def execute(self, args: str, update, context) -> str:
        """
        Execute the skill. Called when user types !example or !ex.
        
        Args:
            args: Everything after the command (!example FOO BAR → args="FOO BAR")
            update: The incoming message (Telegram Update object or similar)
            context: Canopy's context (conversation history, etc.)
        
        Returns:
            Response string to send back to user
        """
        return f"Hello! You said: {args}"
```

### 2. Register Your Skill

Add your skill to `skills/registry.py`:

```python
from skills.example_skill import ExampleSkill

SKILL_REGISTRY = {
    "example": ExampleSkill(),
    # ... other skills
}
```

### 3. Test It

Create `tests/test_my_skill.py`:

```python
import pytest
from skills.my_skill import MySkill

@pytest.mark.asyncio
async def test_basic_execute():
    skill = MySkill()
    result = await skill.execute("test args", None, None)
    assert "expected output" in result
```

Run: `pytest tests/test_my_skill.py -v`

### 4. Document It

Add a docstring to your skill class explaining what it does and what commands it supports:

```python
class MySkill(Skill):
    """
    My skill does cool things.
    
    Commands:
        !mything <arg1> <arg2>    — Does thing 1
        !myfeature                — Does feature Y
    """
```

## Adding a Research Source

The research engine currently uses DuckDuckGo for web searches. Here's how to add a different source.

### 1. Create a New Source Module

File: `tools/research_source.py`

```python
import logging

logger = logging.getLogger(__name__)

class ResearchSource:
    """Base class for research sources."""
    
    async def search(self, query: str, limit: int = 5) -> list[dict]:
        """
        Execute a search query.
        
        Returns:
            List of dicts: {"url": str, "title": str, "snippet": str}
        """
        raise NotImplementedError

class WikipediaSource(ResearchSource):
    async def search(self, query: str, limit: int = 5) -> list[dict]:
        # Implementation here
        pass
```

### 2. Integrate with Research Engine

In `core/research_engine.py`, add:

```python
from tools.research_source import WikipediaSource

# In ResearchEngine.__init__:
self.sources = {
    "duckduckgo": DuckDuckGoSource(),
    "wikipedia": WikipediaSource(),
}

# In conduct_research:
results = await self.sources["wikipedia"].search(query, limit=3)
```

### 3. Test It

```python
@pytest.mark.asyncio
async def test_wikipedia_search():
    source = WikipediaSource()
    results = await source.search("Python programming")
    assert len(results) > 0
    assert "url" in results[0]
```

## Code Review Standards

When submitting a PR or reviewing AI-generated code:

### 1. Docstrings

Every function needs a docstring explaining **WHAT** it does and **WHY** it exists:

```python
# ✓ Good
def calculate_tokens(text: str, model: str) -> int:
    """
    Estimate token count for text on a given model.
    
    This is used by the complexity judge to decide which model tier
    to dispatch a task to. Underestimates can cause budget overruns;
    overestimates waste money. We use the official tokenizer for
    accuracy.
    
    Args:
        text: The text to tokenize
        model: The model name (e.g., "claude-sonnet-4-6")
    
    Returns:
        Estimated token count
    """

# ✗ Bad
def calculate_tokens(text, model):
    # Calculates tokens
    return count
```

### 2. Security Comments

Any code that executes shell commands, writes files, or fetches from the web must have an inline comment explaining the safety measure:

```python
# ✓ Good
def execute_shell(cmd: str) -> str:
    # SAFETY: Check if cmd is in the allowlist before executing
    if cmd not in SHELL_ALLOWLIST:
        return "Command not allowed"
    subprocess.run(cmd, shell=False)  # shell=False prevents injection

# ✗ Bad
def execute_shell(cmd: str) -> str:
    subprocess.run(cmd, shell=True)  # DANGER: Allows injection!
```

### 3. No Orphan TODOs

Every TODO must be linked to a GitHub issue or resolved:

```python
# ✓ Good
# TODO(issue #42): Add retry logic for failed API calls

# ✓ Also good
# FIXME: Remove this hack once we have a proper webhook system
# See: https://github.com/our-org/canopy-seed/issues/88

# ✗ Bad
# TODO: This is a placeholder
```

### 4. Test Coverage

If you add a function, add a test. If you modify a function's behavior, update its test.

```bash
pytest tests/ -v  # Should pass
pytest tests/ --cov=core/  # Check coverage (aim for >80%)
```

## Reporting Issues

Found a bug? Open an issue on GitHub with:

1. **Clear title:** "Snapshots fail when project name has spaces"
2. **Reproduction steps:**
   - Step 1
   - Step 2
   - What happens (the bug)
   - What should happen (expected behavior)
3. **Environment:** Windows/Linux, Python version, LM Studio vs. Claude, etc.
4. **Logs:** If possible, attach `logs/canopy.log` (redact any sensitive API keys)
